# The "Brain" (OpenDaylight) is successfully controlling the "Body" (Mininet switches).

* you need to provide the "Proof of SDN" for your practical submission. Here are the three things you should do right now to finish the lab.


1. View the Flow Tables (The Southbound Proof)

- The pings worked because OpenDaylight pushed "Flow Rules" to the switches. 
- You should see these rules to prove the controller is doing its job.
  In the Mininet terminal, run:


mininet> dpctl dump-flows


* What to look for: You’ll see a bunch of text showing priority, n_packets, and actions=output. 
* This is the literal "instruction manual" the controller sent to the switches.

## ``dptcl dump-flows``  cmd can throw error cuz of OpenFlow version mismatch. 
to solve that error do below thing.



1. View the Flows (Correct Command)
Run this inside your Mininet prompt:


		mininet> sh ovs-ofctl -O OpenFlow13 dump-flows s1



(This targets switch s1. You can change it to s2 or s3 to see their specific rules).

2. What you are looking at (The Logic)

--> When you run that above cmd , you'll see lines with cookie, duration, and actions.

--> The Controller in Action: Since you used odl-l2switch-switch, OpenDaylight saw the first ping, learned which MAC address is on which port, and pushed an "Output" action to the switch.

--> Southbound Interface: This is the perfect example of the Southbound API working. The controller uses OpenFlow 1.3 to program the "dumb" switches.
-----------------------------------------------------------------------

2. Check the Topology via REST (northbound proof)

- One of the core features of OpenDaylight is the Northbound API. 
- You can ask the controller for the network map using a web request.
- Open a third terminal window and run this curl command:

> curl -u admin:admin http://localhost:8181/restconf/operational/network-topology:network-topology

--> What to look for: A massive wall of JSON text. 
This proves that an external application can "read" the network state from the controller.

--> If you get a JSON response showing node-id, s1, s2, and s3, you have completed the full SDN loop:

1. Data Plane: Mininet created the hosts.

2. Southbound: OpenFlow connected switches to the controller.

3. Control Plane: OpenDaylight learned the path and enabled pings.

4. Northbound: RESTCONF shared the status with the outside world.

--------------------------------------------------------------------------


3. Test Network Agility (The "Dynamic" Proof)

- SDN is meant to be flexible. Let's break the network and see OpenDaylight react.
# Bring a link down: 

		mininet> link s1 s2 down


# Try to ping: 

		mininet> pingall                    (You’ll see some hosts can no longer talk).



# Bring it back up: 

		mininet> link s1 s2 up


# Ping again: 

		mininet> pingall   (Everything should work again. This proves the controller dynamically manages the state).
